    Metadata

    Metadata

    In addition to the content data, the paratext of research data contains different types of additional metadata. The Research Data Alliance (RDA),6 and OpenAIRE5 define metadata as

    data-providing information that enables data to be found, tracked and (re)used. Metadata can therefore include ‘contact information, geographical locations, details about units of measure, abbreviations or codes used in the dataset, instrument and protocol information, survey tool details, provenance and version information and much more’.

    This list of paratext data will be regularly discussed at workshops and expert consultations organised by the NFDI4SD. The metadata need to satisfy the core standards of, for example, Dublin Core, DataCite or OpenAIRE. We intend to adopt the DataCite Metadata Schema (see Datacite). From the experience we have gained in data aggregation, we believe that large sets of metadata and standardised ontologies only play a subordinate role in the small disciplines. A standardised terminology will be closely controlled during the data preparation process.

    Ontologies/International Committee for Documentation (CIDOC)/Reference dates

    The NFDI4D will optionally provide a rich portfolio of machine interpretable ontologies that are comparable to Achres, such as the CIDOC Conceptual Reference Model (see CIDOC). The bottom-up model of semantic frameworks and their inferential processing will enable the projects to review their own application areas. The machine learning algorithms based on this model must always show their quality metrics against the annotated reference data set. Ideally, these reference data are published, manually selected and then checked by the researchers.

    The following considerations will be taken into account:

    • JSON-LD, which is a lightweight Linked Data format that allows a standardised API for complex, hierarchical data structures, is the best format for encoding paratext data. Its standards are ideal for an interoperable API, Representational State Transfer (REST) web services and a web interface.3 Transformations into other formats and standards are also planned.
    • Core metadata, as required by Zenodo and generic data repositories, will be included in the set of metadata.
    • Data provenance will be included in the paratext so that a data dependency graph can be constructed. The graph will enable the NFDI4SD’s registry to manage data dependencies and effectively navigate through data repositories.
    • The Bavarian State Library in Munich, which is an important content provider for digital resources, has assigned metadata tags to more than 2.5 million digital objects; it also has a number of RDF Schema and XML Schema structured metafiles. These schemas will be used transparently to process our researchers’ research data. We intend to ensure data and metadata coherency.
    • Meaningful data APIs will be built that take API features into consideration (see API).
    • Flexible data mapping between user-specific terminological and data standards systems will be provided by the NFDI4SD’s machine learning tools.
    • Well-documented data, their origin, data units, the definitions of terms and standards are recommended and will be assisted by the appropriate tools using research feedback from the specific research community.
    • A public review on a new Data Documentation Initiative (DDI) product – the Structured Data Transformation Language (SDTL;Structured Data Transformation (SDTL) – that addresses the issue of data provenance is currently taking place.4 A proposal is is being drawn up.

    NFDI4SD Standards-Companion

    The proven and widely used services that include Zenodo, reference data services, programming languages services and ontologies presuppose a number of key global standards and norms for data use. A range of computational tools have been developed to help researchers choose an appropriate set of standards and ontologies for their work.

    A selection of widely used metadata standards, both general and domain specific, includes:

    • Dublin Core – a basic, domain-agnostic and easily used and implemented metadata standard

    • DDI (Data Documentation Initiative) – an international standard for describing the data (including survey data) produced in the social, behavioural, economic and health sciences

    • EEML (Ecological Metadata Language) – developed for the disciplines of ecology

    • ISO 19115 and FGDC-CSDGM (Federal Geographic Data Committee’s Content Standard for Digital Geospatial Metadata) - defines the schema required for describing geospatial information

    • MINSEQE (Minimal Information about a high-throughput SEQuencing Experiment – genome sequencing guidelines

    • FITS (Flexible Image Transport System) – a digital file standard that includes structured, embedded metadata for astronomical data sets

    • MIBBI (Minimum Information for Biological and Biomedical Investigations) - a set of guidelines for reporting data derived in the biosciences

    Where no appropriate, formal metadata standard exists, it is recommended that researchers create a ‘readme’ file.

    The following resources contain additional information:

    • DCC Metadata Standards

    • RDA Metadata Directory

    Other standards and ontologies include:

    • Zenodo API
    • PID systems
    • JSON-LD
    • UTF 8
    • CIDOC2
    • Schema.org

    Researchers are not expected to get to grips with all these standards or to be able to judge their advantages and disadvantages.

    Rather, the individual disciplines need to develop or use their own specific terminologies, so that the research data distributed over various contexts of origin gradually acquire a conceptual vagueness, which creates difficulties when using simple terminology uncritically. Mees, Thiery and Unold have convincingly pointed out cases of provincial Roman archaeology and many others to deal with vague concepts rather than taxonomy.1 Linked Open Data (LOD) therefore needs to go beyond the sole use of ontologies and move from the possibilities of machine learning to the tried and tested use of vague concepts.

    In order that researchers can use the metadata standards as a service, the NFDI4SD will develop a Standards-Companion. Similar to a Companion on Legal Standards, which is also to be established by the NFDI4SD, the Standards Companion will, through the use of machine learning, classify the registered research data according to the standards that are being followed and will directly generate an automated report on the standards used, their reference pages and additional literature via the Virtual research bench. Alternative concepts of standards will also be outlined. The Standards Companion will constantly update its own database and in this way will ensure that the research data are harmonised. The principal data sources for the Standards Companion are:

    • Formal models of the standards
    • Labelled data for machine learning

    Provenance

    Data provenance will become an integral part of the research data’s paratext. Provenance information will be gathered to construct a dependency graph representing the dependencies that research data objects have on each other as described in the narrative of computational notebooks. Each object must be assigned a unique identifier and a reference in a research publication in order that the dependencies between them can be established. The NFDI4SD’s registry will build up a large dependency graph for research data that will support the cataloguing and reuse of the research data.
